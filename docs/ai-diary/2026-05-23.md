# AI diary — 2026-05-23

## Session focus

**Geocoder performance (Phase 1 → 2 → 3)** and **media-detail Internet search UX** (pulse, back/forward history, list-level dedupe). End state: bounded Nominatim per PostGIS media cluster, city-hint preserved, cluster RPC cached per project/radius.

## Situation we had

| Layer | Problem |
| ----- | ------- |
| **Nominatim** | Global search + client-side distance filter → weak labels, multi-city projects (e.g. Vienna + Paris) where one region wins in ranking |
| **Call budget** | Phase 1 removed suffix probes / unconstrained retry; still needed structured city-hint and short-prefix `layer=address` fix (`denis` vs `denisgasse`) |
| **UI** | Internet dropdown: loading pulse, history stacks; user wanted **list equivalence** (two result lists are the same iff the **set of addresses** matches — not row-by-row dedupe inside one list) |
| **Phase 3** | Need cluster viewboxes from project GPS without touching every callsite; plan had wrong table name (`media` vs `media_items`) and wrong tuning key (`maxDistanceForInternetResults` vs `contextDistanceMaxMeters`) |

## What went well in communication

### From the user

- **Correcting the mental model in one sentence** — e.g. list uniqueness = same multiset of addresses across the whole dropdown, not deduping rows inside one list. That unblocked history logic immediately.
- **Plan approval with a precise SQL delta** — “eps stays `km/111`; fix **only** viewbox `lon_pad` with `cos(lat)`” — no re-litigation of DBSCAN, no scope creep.
- **Explicit implementation choice** — WorkspaceViewService inject for project scope (option C): avoids expanding allowed-file list and still covers media-detail.
- **Attached plan + “do not edit plan file”** — stable contract while iterating implementation details in chat.
- **Hard stops when needed** — STOP during unrelated renames; “simple fix gate” for small behavioral bugs.

### From the agent (when it worked)

- **Phase 0 report** in the plan (wrong paths, `rankGeocoderCandidates` slice bug, tuning field retention) before coding.
- **Pre-filled codebase audit** (PostGIS, GIST, `media_items`, existing Edge `bounded`) so implementation matched Feldpost, not generic prompt text.
- **AskQuestion** for project-scope source (media-detail vs map vs WorkspaceViewService) instead of guessing.

## What went wrong (agent / process)

| Mistake | Cost | Fix pattern |
| ------- | ---- | ----------- |
| Misread “dedupe” as **within-list** row dedupe | Wrong history pushes, user had to restate list-level rule | Ask: “Same **list** or same **row**?” before implementing equivalence |
| Assumed prompt column/table names (`media`, `maxDistanceForInternetResults`) | Plan would have shipped wrong SQL | Always map prompt → `database-schema.md` + `search-tuning.types.ts` in Phase 0 |
| Long discovery on UI when user already stated invariant | Tokens + frustration | **Simple fix gate:** if rule fits in one sentence, implement; don’t offer three architectures |
| Summarized plan approval without encoding `lon_pad` until user sent SQL snippet | Risk of shipping equator-only padding | On “approved + one point”, paste that point into plan **and** migration in same turn |

## How to improve prompting (for humans writing agent prompts)

### Do (high leverage)

1. **Name the invariant first** — “Two lists are equal iff …” before UI or algorithm details.
2. **Allowed file list** — reduces scope creep; pair with “ask if you need X outside list”.
3. **Phase 0: STOP on blockers** — PostGIS missing, wrong columns → report, don’t invent JS fallback.
4. **Locked decisions inline** — “eps unchanged; only viewbox padding uses cos(lat)”.
5. **Precedence rules** — e.g. “clusters override map viewportBounds for Nominatim”.
6. **Call-budget table** — before/after per scenario; agents verify against it.
7. **One approval channel** — “green light on plan” vs “change code now”; avoids half-implemented plan drift.

### Avoid

- Mixing **plan iteration** and **execute** in the same message without saying which.
- Generic table names (`media`) when the repo uses `media_items`.
- Assuming tuning UI labels match code property names — cite `search-tuning.types.ts`.

### Prompt skeleton (geocoder / search pipeline)

```markdown
## Invariant
<one sentence>

## Must not regress
- <bullet list from prior phases>

## Allowed files
- path → action

## Phase 0 (read-only)
- SQL checks + codebase grep results; STOP if …

## Locked decisions
1. …

## Verification
- vitest paths, ng build, manual scenario
```

## How agents can ask to clear things up (more often)

**User preference (2026-05-23):** Not “1–2 precise questions” — ask **enough** that behavior is unambiguous. Batch questions in one round; don’t drip single questions across many turns.

When any trigger below applies, add it to a **clarification list** and ask before coding (don’t wait for the third wrong implementation):

| Trigger | Example question |
| ------- | ---------------- |
| **Equivalence / dedupe / merge** | “Same whole list, or duplicate rows inside one list?” |
| **Scope of “viewport”** | “Map pan bounds, project media clusters, or marker-only?” |
| **Precedence** | “If both map viewport and clusters exist, which wins?” |
| **Empty vs fallback** | “0 clusters → global search or keep map viewbox?” |
| **Identity key** | “Dedupe by label only, or label + lat/lng?” |
| **UI history** | “Skip push when equivalent, or still update visible list?” |
| **File boundary** | “OK to add `activeProjectId` in media-detail context (not in allowed list)?” |

### Before a multi-file feature

1. Restate **invariant** in your own words; ask “correct?” if non-trivial.
2. List **all** ambiguities from plan/spec/code — ask about each unresolved one.
3. State **what you will not touch** (allowed list + “no RxJS”, “no new components”).
4. If the user corrects you once, **update the spec or plan section** they care about — don’t only fix code.

### When the user is terse or frustrated

- Shorter replies; one hypothesis; then **the remaining open questions** (can be several, numbered).
- Prefer **AskQuestion** when options are discrete (WorkspaceViewService vs media-detail — worked well).
- Do **not** interpret frustration as “ask less” — often it means the agent should have asked **more** before building the wrong thing.
