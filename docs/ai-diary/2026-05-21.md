# AI diary — 2026-05-21

## Session focus

Upload panel / settings **hlmSwitch** thumb: hover nudge, track darken on hover, smooth slide easing, vertical alignment in multi-line toggle rows, zoom-stable positioning, design tokens — and a long detour caused by layering fixes on top of each other without re-checking the baseline.

## What we shipped (end state)

- **`switch-thumb-hover.scss`** — thumb slide via `transform` + tokenized motion (`--motion-duration-fast * 1.8`, segmented-switch ease); hover track darken via `color-mix` on `--input` / `--primary`; hover nudge via `--spacing-1`; thumb geometry uses spacing grid `calc(0.25rem * 5)` inside the track padding box (`top`/`left: 0`).
- **`switch-variants.ts`** — shadcn track `inline-flex h-6 w-11 items-center`; thumb `h-5 w-5`; no `ms-auto` / no margin animation (that was the “weird curve”).
- **Row wrappers** — `.upload-panel__location-mode-switch` and `.settings-overlay__switch`: `flex: 0 0 auto`, `align-self: center`, `line-height: 0`, **no `display: flex` on the wrapper**.
- **Auto location icon** — `align-self: center` (was `flex-start`).

## Root cause of the vertical offset

The thumb is **`position: absolute`** inside `[hlmSwitch]`. A **parent wrapper** had `display: flex` (added during debugging). That created a **second flex formatting context** and disturbed how the track + thumb laid out. Disabling that wrapper rule in DevTools fixed alignment immediately.

**Lesson:** The track’s own `inline-flex` (shadcn) is enough. The wrapper only needs to be a **flex child** of the row (`flex: 0 0 auto`, `align-self: center`), not a flex container itself.

## What went wrong (agent mistakes)

| Mistake | Why it hurt |
| -------- | ------------ |
| Added `display: flex` on the switch wrapper “to help centering” | Extra flex context; fought absolute thumb positioning |
| Chased vertical alignment with `top: 2px`, `translateY(-50%)`, `ms-auto`, `100cqw`, `left` transitions | Each patch addressed a symptom; stacked contradictions |
| Mixed `px`, `rem`, `%`, and `100cqw` for the same travel | Zoom / rounding drift; user correctly suspected unit mismatch |
| Removed thumb visibility once by sizing track on a `<span>` without `display` | Full rebuild instead of minimal revert |
| Did not stop after user said “don’t rebuild everything” | Trust erosion; more time lost |

## How to avoid this next time (technical protocol)

1. **One geometry owner.** Switch layout lives in **one place**: either CVA (shadcn) + a small SCSS file for hover/slide only — not both fighting with absolute + flex + margin.
2. **Before adding a wrapper style, ask:** “Is this element already a flex *child*? If yes, what does `display: flex` on it add?” If the answer is vague, **do not add it**.
3. **Absolute thumb ⇒ parent is `position: relative` on the track only.** Wrappers: `line-height: 0`, no flex display, no extra positioning.
4. **Prefer token grid for size/travel:** `calc(0.25rem * N)` aligned with Tailwind `h-6` / `w-11` / `h-5`; avoid `calc(100% - Xrem)` for horizontal travel when `%` and `rem` mix at zoom.
5. **Animate one property for slide:** `transform: translateX(...)` only — never `margin` + `ms-auto` for the same motion (weird easing).
6. **Verify in DevTools in this order:** (a) thumb box centered in track, (b) toggle row height change, (c) browser zoom 90% / 110%, (d) hover/focus not pinned by `:focus-within` on parent `<button>`.
7. **When the user disables a rule in DevTools and says “it works”** — treat that as the lead signal; revert the last additive change, do not add another layer.

## How our communication can work better

### What helped today

- User screenshots and **“when I turn off flex it works”** — concrete, reproducible.
- Clear product intent: shadcn-like switch, nice **curve** on toggle, darker track on hover, stay close to design tokens.
- Pushback: **“thumb size was perfect”**, **“stay on shadcn”**, **“don’t rebuild everything”** — bounded the solution space.

### What to ask earlier (agent)

Before touching switch/toggle CSS, ask in one short block:

1. Is misalignment **inside the pill** or **the pill in the row**?
2. Does it happen **only when the row is tall** (two-line label)?
3. Does it change with **browser zoom**?
4. OK to only edit `switch-thumb-hover.scss` + row wrapper, or is a new pattern required?

### What to state before each change (agent)

- **Hypothesis** (one sentence).
- **Files touched** (max 1–2).
- **What we will not change** (e.g. thumb size, shadcn track classes).
- **How to verify** (reload, toggle on/off, hover row, zoom).

### What to do when stuck after 2 failed attempts

**Stop.** Summarize what was tried, what the user sees, and offer **two options** (e.g. “revert to ms-auto snap” vs. “tokenized transform slide”) — do not auto-pick a third architecture.

### What helps from the user (optional, not required)

- One screenshot + “inside track vs row” if alignment looks off.
- DevTools hint like today: **which rule disabled fixes it**.
- “Minimal diff only” when frustration shows — agent must comply.
